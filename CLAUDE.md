# CLAUDE.MD — 프로젝트 Claude Code 행동 지침서

이 문서는 Claude Code가 본 프로젝트에서 **어떤 규범을 따르고, 어떤 워크플로우로 작업을 수행해야 하는지**를 정의합니다. 실질적인 구현 내용은 Plan 모드에서 제안 → 승인 후 실행하는 흐름을 따릅니다.

## 2\. 필수 참조 문서

Claude Code는 작업 착수 전 반드시 관련 문서를 읽고 내용을 파악해야 합니다.

| 문서 | 경로 | 참조 시점 |
| :---- | :---- | :---- |
| **데이터 구조 가이드** | `Documents/데이터 구조 가이드/` | 데이터 모델·SO·테이블 설계 및 구현 시 **반드시** 참조 |
| 기획 컨셉 및 게임플레이 개요 | `Documents/기획 컨셉 및 게임플레이 개요.md` | 게임 전체 방향성·루프·핵심 메커니즘 확인 시 |
| 각 씬(레이어)에 대한 규정 | `Documents/각 씬(레이어)에 대한 규정.md` | 레이어별 상세 규칙·보상 체계·전투 흐름 구현 시 |
| VN JSON 데이터 구조 | `Documents/비주얼 노벨 — JSON 데이터 처리 구조 & 전체 필드 정리.md` | VN 연동·데이터 참조 시 |

### 데이터 구조 가이드 준수 규칙

⚠️ 최우선 규칙: 데이터 모델, ScriptableObject, Enum, 테이블 구조를 새로 만들거나

수정할 때는 반드시 Documents/데이터 구조 가이드/ 내 해당 문서를 먼저 확인하고,

해당 문서에 정의된 속성명·타입·제약조건을 그대로 따른다.

\- 가이드에 정의되지 않은 필드를 임의로 추가하지 않는다.

\- 가이드의 네이밍과 다른 이름을 사용하지 않는다.

\- 가이드에 모호한 부분이 있으면 가정하지 말고 질문한다.

\- 가이드에 없는 새 데이터 구조가 필요하다면, Plan 모드에서 먼저 제안한다.

## 4\. 코딩 컨벤션

### 4.1 네이밍

| 대상 | 규칙 | 예시 |
| :---- | :---- | :---- |
| 네임스페이스 | PascalCase, 프로젝트 접두사 | `{ProjectName}.Battle` |
| 클래스 / 구조체 | PascalCase | `TurnManager`, `CardData` |
| 인터페이스 | I \+ PascalCase | `IDamageable` |
| public 메서드 | PascalCase | `TakeDamage()` |
| private 메서드 | PascalCase | `CalculateSpeed()` |
| public 필드 | PascalCase | `MaxHealth` |
| private 필드 | \_camelCase | `_currentEnergy` |
| \[SerializeField\] private | \_camelCase | `_deckCapacity` |
| 상수 | UPPER\_SNAKE 또는 PascalCase | `MAX_PARTY_SIZE` |
| 이벤트 | On \+ PascalCase | `OnTurnStarted` |
| bool | Is/Has/Can 접두사 | `IsGrounded` |

### 4.2 스크립트 템플릿

using UnityEngine;

namespace {ProjectName}.{Layer}

{

    /// \<summary\>

    /// {클래스 설명 \- 반드시 XML 문서 주석 작성}

    /// \</summary\>

    public class {ClassName} : MonoBehaviour

    {

        \#region Serialized Fields

        \[Header("Settings")\]

        \[SerializeField\] private float \_value \= 1f;

        \#endregion

        \#region Private Fields

        private bool \_isInitialized;

        \#endregion

        \#region Public Properties

        public bool IsInitialized \=\> \_isInitialized;

        \#endregion

        \#region Events

        public event System.Action OnCompleted;

        \#endregion

        \#region Unity Lifecycle

        private void Awake() { }

        private void OnEnable() { }

        private void OnDisable() { }

        \#endregion

        \#region Public Methods

        \#endregion

        \#region Private Methods

        \#endregion

    }

}

### 4.3 필수 규칙

\- 모든 public 멤버에 XML 문서 주석(///) 작성

\- \[SerializeField\]를 사용하고 public 필드 노출 금지

\- \#region으로 코드 영역 구분

\- 네임스페이스 필수 사용 (레이어별 하위 네임스페이스)

\- var 사용은 타입이 명확한 경우만 허용

\- 매직 넘버 금지 → const 또는 \[SerializeField\]로 대체

\- async/await 사용 시 UniTask 기반 (또는 Awaitable, Unity 6+)

## 5\. 금지 사항

\- GameObject.Find(), FindObjectOfType() 등 런타임 탐색 사용 금지

\- 싱글톤 남용 금지

\- Update()에서의 GC Alloc 유발 코드 금지 (string 연결, LINQ, new 등)

\- Resources 폴더 사용 금지

\- 게임 수치(대미지 배율, 코스트 등)의 매직 넘버 사용 금지

\- Documents/데이터 구조 가이드/ 에 정의된 속성을 임의로 변경하거나 누락하는 것 금지

\- 기존 VisualNovel/ 코드의 구조·네이밍·패턴을 무시하고 새로 작성하는 것 금지

## 9\. 작업 워크플로우

### 9.1 기본 원칙

Claude Code는 코드를 즉시 작성하지 않는다.

모든 작업은 \[분석 → Plan 제안 → 승인 → 실행\] 흐름을 따른다.

### 9.3 작업 요청 형식 (사용자용)

사용자는 아래 형식으로 요청하면 가장 효율적입니다.

#### 기능 구현

\#\# 작업: {기능 이름}

\- \*\*유형\*\*: feat / fix / refactor / perf / test

\- \*\*레이어\*\*: Meta / Stage / Battle / VisualNovel / Core / Data

\- \*\*설명\*\*: {구현할 내용}

\- \*\*의존성\*\*: {참조해야 할 기존 클래스/인터페이스}

\- \*\*제약 조건\*\*: {있다면}

#### 버그 수정

\#\# 버그: {증상 요약}

\- \*\*레이어\*\*: Meta / Stage / Battle / VisualNovel

\- \*\*재현 경로\*\*: {발생 상황}

\- \*\*기대 동작\*\*: {올바른 동작}

\- \*\*실제 동작\*\*: {잘못된 동작}

\- \*\*에러 로그\*\*: {있다면}

#### 리팩터링

\#\# 리팩터링: {대상}

\- \*\*현재 문제\*\*: {왜 필요한지}

형식을 정확히 따르지 않아도 됩니다. 자유롭게 요청하면 Claude가 위 형식으로 정리하여 Plan을 제안합니다.

## 10\. 응답 규칙

1\.  코드를 작성하기 전에 반드시 Plan을 먼저 제안하고 승인을 받는다.

2\.  데이터 구조를 설계·구현할 때는 Documents/데이터 구조 가이드/를 반드시 먼저 읽는다.

3\.  기획 규칙 관련 구현 시 Documents/ 내 해당 기획 문서를 확인한다.

4\.  새 파일 생성 시 디렉터리 구조(§3)를 따른다.

5\.  코딩 컨벤션(§4)을 준수한다.

6\.  금지 사항(§5)에 해당하는 코드를 작성하지 않는다.

7\.  성능 가이드라인(§6)을 고려한다.

8\.  허용된 서드파티 패키지(§7)만 사용한다.

9\.  Assets/VisualNovel/ 수정 시 기존 코드의 구조·패턴·네이밍을 먼저 파악하고 따른다.

10\. 변경 사항은 한국어로 간결하게 요약한다.

11\. 확실하지 않은 사항은 가정하지 말고 질문한다.

12\. 에디터 전용 코드는 \#if UNITY\_EDITOR로 감싼다.
